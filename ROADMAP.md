## 3. P2 — 优化（v3.5.1 维护 Backlog）

### 3.1 P2-1: Footer 字体加载导致 CLS 0.3756

| 项 | 值 |
|---|---|
| **为什么做** | Core Web Vitals CLS 不达标（0.3756 > 0.1 阈值），可能影响搜索引擎排名和用户体验评分 |
| **预计影响** | CLS 从 0.375 降低到 < 0.1，提升 Core Web Vitals 评分 |
| **风险** | 低 — font preload 是标准做法，不影响视觉 |
| **优先级** | 中（用户体验 + SEO） |
| **推荐方案** | font preload — 在 `index.html` `<head>` 添加 `<link rel="preload" as="font">` 预加载关键字重 |
| **修改范围** | `index.html`（仅添加 preload 标签） |
| **处理时机** | v3.5.1 维护期，需用户批准 |

### 3.2 P2-2: Amber accent + white 文字对比度 3.19:1

| 项 | 值 |
|---|---|
| **为什么做** | WCAG AA 合规要求 normal text ≥ 4.5:1，当前 3.19:1 不达标 |
| **预计影响** | 对比度提升到 ≥ 3:1（large text 阈值），符合 WCAG AA large text 标准 |
| **风险** | 极低 — `font-weight: 600` 仅影响按钮文字粗细，不影响布局 |
| **优先级** | 低（baseline 设计决策，WCAG AA 非强制） |
| **推荐方案** | `font-weight: 600` — 按钮文字加粗到 600，触发 large text 阈值（≥ 14px bold） |
| **修改范围** | `src/styles/tokens.css` 或 `src/styles/global.css`（仅按钮 font-weight） |
| **处理时机** | v3.5.1 维护期，需用户批准 |

### 3.3 P2-3: FCP 线上复测（已自愈，仅记录）

| 项 | 值 |
|---|---|
| **为什么做** | 无需处理（已自愈） |
| **预计影响** | 无 |
| **风险** | 无 |
| **优先级** | —（已自愈） |
| **结论** | FCP 问题为 local server 环境噪声，线上 Vercel CDN + Edge Network 加速后已达标（< 1500ms ✅） |
| **处理时机** | 无需处理 |

## 5. v3.5.1 维护期处理计划

### 5.1 处理顺序

| 顺序 | 问题 | 优先级 | 预计工作量 | 修改范围 |
|---|---|---|---|---|
| 1 | P2-1 Footer CLS | 中 | 0.5h | `index.html`（font preload） |
| 2 | P2-2 Amber 对比度 | 低 | 0.5h | `tokens.css` 或 `global.css`（font-weight） |
| 3 | P2-3 FCP | — | 0 | 无需处理 |

### 5.2 处理原则

- **最小修改**：仅触碰必须改的地方，不重构无关代码
- **不扩大 Scope**：不新增组件 / 依赖 / Design Token / 颜色 / 字体
- **可回滚**：每个修改都应能独立回滚
- **完整验证**：每项修复后执行 typecheck + build + Playwright + 线上验证
- **用户批准**：所有修改需用户明确批准后才能动工

### 5.3 v3.5.1 发布条件

- ✅ 至少处理 P2-1（Footer CLS）
- ✅ 通过完整验证（typecheck + build + Playwright + 线上 Core Web Vitals）
- ✅ CLS 从 0.375 降低到 < 0.1
- ✅ 用户验收通过

### 5.4 v3.5.1 发布流程

1. 用户批准进入 v3.5.1 维护期
2. 按顺序处理 P2-1 → P2-2
3. 每项修复后执行完整验证
4. 生成 v3.5.1 Review Report
5. 用户验收
6. Git commit + tag `v3.5.1` + push origin
7. Vercel 部署验证 + 线上 Core Web Vitals 验证
8. 更新交接文档（HANDOFF / PROJECT_STATUS / TECHNICAL_DEBT / ROADMAP）

## 6. 新 Roadmap 批准流程

如需启动 v3.6+ 或任何新功能 / 大重构，需遵循以下流程：

1. **用户提出新需求** — 明确范围、目标、优先级
2. **AI 分析影响** — 评估对冻结清单、SSOT、组件配额、bundle 预算的影响
3. **用户批准新 Roadmap** — 明确允许新增组件 / 依赖 / Design Token 等
4. **创建实施计划** — 类似 `Portfolio_v3.5_IMPLEMENTATION_PLAN.md`
5. **创建验收标准** — 类似 `Portfolio_v3.5_IMPLEMENTATION_READINESS.md`
6. **Phase 化执行** — 每个 Phase 完整生命周期：开发 → 验证 → Review → 文档 → Commit → Push → Report
7. **每个 Phase 后停止** — 等待用户批准下一 Phase

**未经用户明确批准，不得擅自启动新 Roadmap。**
