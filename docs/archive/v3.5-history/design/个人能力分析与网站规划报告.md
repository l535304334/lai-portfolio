## 项目1：医学考试刷题系统（题库）

### 基本信息

| 维度 | 内容 |
|------|------|
| **项目名称** | 医学考试刷题系统 |
| **项目类型** | 纯前端单页应用（SPA） |
| **项目背景** | 为医学考试（中医/临床）备考设计，覆盖体格检查、实验室检查、问诊诊断、心电图、共选五大科目 |
| **项目目标** | 提供一个"打开即用"的刷题工具，支持日常练习、错题复习、模拟考试、间隔复习 |
| **技术栈** | HTML5 + CSS3（自定义设计令牌系统）+ Vanilla JavaScript（ES5+）+ Chart.js v4.4.4 |
| **开发环境** | 无构建工具、无包管理器、无服务器 — 纯浏览器运行 |
| **项目规模** | 单文件 14,563 行（含 ~5,944 道题目数据、~2,600 行 JS 逻辑、~5,870 行 CSS、~5,500 行 Chart.js） |
| **完成时间** | 2026年6月-7月（V1→V9 迭代） |

### 技术亮点

1. **SM-2 间隔复习算法**：完整实现 SuperMemo SM-2 算法（EF 因子、间隔调度、每日复习队列），对于一个独立 HTML 应用而言有技术含量
2. **CSS 设计令牌系统**：70+ 语义化 CSS 自定义属性，暗色模式仅覆盖 ~20 个颜色变量，架构清晰
3. **异步索引构建**：全局搜索通过 `requestIdleCallback` 非阻塞构建索引，考虑到了性能
4. **localStorage 配额保护**：内置配额监控 + 自动淘汰最旧快照，使用 `navigator.storage.estimate()` API
5. **复合模式过滤**：错题/复习/收藏三种模式可叠加使用，用集合交集逻辑过滤
6. **完整的数据管理**：快照备份/恢复（最多10个版本）、JSON 导入/导出/合并、自定义题目管理
7. **版本迭代意识**：从 V7 到 V9 有明确的版本规划和优化方案文档

### 普通之处（需要优化描述）

- **架构层面**：单文件 14,000+ 行是"能用但不专业"的架构。可描述为"快速原型验证"，面试时应主动说明如果需要工程化会如何拆分
- **数据处理**：用 `indexOf` 做全文搜索而非倒排索引，对于 5,944 道题数据量足够，但需要说明这是"YAGNI 决策"
- **无版本控制**：Git 仓库无任何 commit，需解释开发过程
- **无测试**：缺少任何形式的自动化测试

### 基本信息

| 维度 | 内容 |
|------|------|
| **项目名称** | 两地书 Love |
| **项目类型** | 微信小程序（全栈 Serverless） |
| **项目背景** | 为异地恋大学生情侣设计的日常陪伴工具，灵感源自鲁迅与许广平《两地书》 |
| **项目目标** | 提供书信交流、心情记录、经期关怀、AI 智能助手等陪伴功能 |
| **技术栈** | Taro 4.2 + React 18 + TypeScript 5.3（strict）+ Zustand 4.5 + SCSS |
| **后端** | 微信云开发 CloudBase（26 个云函数 + 22 个数据库集合 + 云存储） |
| **AI 集成** | DeepSeek Chat API（8 个 AI 场景） |
| **外部 API** | 和风天气 API |
| **测试** | Vitest（28 个单元测试） |
| **开发环境** | Node.js + Webpack 5 + ESLint 10 flat config |
| **项目规模** | ~130+ 源文件、23 个页面、14 个共享组件、26 个云函数、~12,000-15,000 行代码 |
| **完成时间** | 2026年6月-7月（含 Phase 0-8 全部任务） |

### 技术亮点

1. **AI 深度集成**：9 个独立 AI 云函数覆盖核心场景（书信润色、诗歌生成、语感检查、备忘提取、科普文章、健康建议、打卡/愿望推荐、周报月报），共享 `callDeepSeek` 模块消除重复代码 — 这是真实的 AI 工程化实践
2. **安全第一的设计**：云函数从 `OPENID` 服务端推导身份，不信任前端传入 userId；字段白名单；创建者所有权校验；Prompt 注入防护；`msgSecCheck` 内容审核
3. **完整的设计系统**：60+ CSS 变量的马卡龙主题、信纸横线纹理（纯 CSS `repeating-linear-gradient`）、粉调阴影、弹性动画曲线
4. **工程化质量**：TypeScript strict 模式 0 error；ESLint 0 error；Zustand 状态管理职责清晰分离；乐观更新模式
5. **合规工程**：完整隐私合规（用户主动勾选协议、引导页多入口、游客模式）
6. **幂等与容错**：定时任务幂等机制、频率限制 DB 故障时 fail-open、单对失败不阻断全局
7. **诚实的技术债务记录**：在 RELEASE.md 中记录了 4 个已知风险和后续路线图
8. **MCP 服务集成**：包含 DeepSeek 和和风天气的 MCP 服务器

### 普通之处（需要优化描述）

- **测试覆盖不足**：28 个单元测试集中在前端工具函数，云函数无测试、无集成测试
- **部分集合客户端直写**：4 个集合仍依赖安全规则而非云函数
- **首页性能问题**：Tab 切换触发 10+ 云函数调用
- **微信生态绑定**：技术栈与微信云开发强耦合，可迁移性有限

## 项目3：江南出行智慧服务平台（jiangnan-travel）

### 基本信息

| 维度 | 内容 |
|------|------|
| **项目名称** | 江南出行智慧服务平台 |
| **项目类型** | 全栈 Web 应用（毕业实习项目） |
| **项目背景** | 南昌大学软件工程 2307 班毕业实习 — 面向江西省的网约车调度平台 |
| **项目目标** | 搭建一个具备并发调度、动态评分、实时通信、AI 助手的出行服务平台 |
| **技术栈** | Spring Boot 3.2.6 + Java 17 + MyBatis-Plus 3.5.7 + MySQL 8.0 + Redis + Redisson 3.32.0 |
| **前端** | Vue 3.4 + Vite 5 + Element Plus 2.7 + Pinia + ECharts 6 + AMap JS API 2.0 |
| **部署** | Docker Compose（MySQL + Redis + Backend + Nginx）+ GitHub Actions CI |
| **测试** | 81 个 Java 单元/集成测试 + 155 个前端测试用例 + Playwright E2E |
| **项目规模** | 218 个 Java 源文件、~65 个 Vue 文件、27 张数据库表、97 个 API 端点、22 个 Controller、30 个 Service |
| **开发周期** | 2026.07.06 - 2026.08.02（4 周实习），2026.06.24 - 2026.07.08 迭代优化 |
| **Git 提交** | 40+ 个 conventional commits |
| **GitHub** | `github.com/l535304334/jiangnan-travel` |

### 技术亮点（核心展示项目）

1. **两阶段分布式锁并发调度**：Redis Redisson 实现 `order:dispatch:{orderId}` 一阶段锁 + `driver:lock:{driverId}` 二阶段 tryLock（500ms 超时）。压力测试验证 100 订单 × 20 司机零重复分配、零死锁。平均调度延迟 <150ms 单订单、<350ms 50 订单批量。**这是可以向导师和面试官深入讲解的核心亮点。**
2. **收敛型动态评分引擎**：静态加权评分（距离 40% + 评分 30% + 空闲 20% + 拒单惩罚 10%）+ 动态反馈学习层（接单 +5、拒单 -10、10 分钟半衰期衰减、边界 [-30, +20]）。5 轮调度接单率从 50% 提升至 70%，司机评分方差降低 50%。**这是"算法设计能力"的强证明。**
3. **形式化订单状态机**：10 个状态、15+ 合法转换路径集中在枚举内 `ALLOWED_TRANSITIONS` Map，每次转换经 `guardTransition()` 校验。18 个单元测试覆盖所有合法/非法路径。**这是"软件工程规范"的强证据。**
4. **关键操作留痕审计追踪**：`t_order_event` 表记录多种事件类型，单条 SQL 即可追踪订单全生命周期。**展示了领域驱动设计思想。**
5. **三层可观测性**：指标层（成功率/延迟/负载分布）+ 健康层（S/A/B/C/D 加权评分）+ 异常层（连续拒单 ≥5 / 失败 ≥3 / 锁竞争 ≥10 检测规则）
6. **AI 集成**：DeepSeek API 通过 OpenAI 兼容 SDK 集成，SSE 流式输出（SseEmitter 打字机效果），旅游知识库注入系统提示词，优雅降级
7. **安全加固**：JWT + Spring Security 三角色鉴权、操作日志密码过滤、JWT 黑名单吊销、速率限制（60 次/分/IP Redis 滑动窗口）、9 个关键安全修复
8. **完整的工程文档**：25+ 份文档覆盖架构、数据库、API、部署、面试准备、实习日志、Bug 分析
9. **DevOps 完整链路**：Docker Compose 四服务编排、GitHub Actions CI、Nginx WebSocket 代理、生产级配置管理

### 面试可直接展示的亮点

> 本项目的亮点已被系统整理在 `docs/interview/` 目录中：
> - **3 分钟/5 分钟自我介绍脚本**
> - **8 个高频面试问答及速查表**
> - **8 个技术深挖亮点**（每题含技术架构、核心逻辑、可展示代码、面试话术）
> - **白板架构图绘制指南**
> - **150 字/100 字简历项目描述**

### 需要补充或说明

- **模拟支付**：支付和短信均为模拟实现，面试时应主动说明
- **无生产部署**：系统未上线，应在网站中展示 Demo 截图或录屏

## 网站定位

### 网站名称建议

| 方案 | 名称 | 理念 |
|------|------|------|
| **推荐** | **「软件工程学生 · 技术成长档案」** | 准确描述定位，适合学术+求职双场景 |

### 核心理念

**不是"个人主页"，是"技术成长档案"：**

- ❌ 不是：一张头像 + 一段自我介绍 + 项目列表（普通简历网站）
- ✅ 是：每个项目展示「遇到了什么问题 → 想了什么方案 → 怎么实现的 → 踩了什么坑 → 学到了什么 → 下次会怎么做」

**差异化定位：**

> "我不是一个'会用 Spring Boot 的学生'。我是一个能独立分析并发调度问题、设计分布式锁方案、写形式化状态机、并用压力测试验证的学生。这个网站就是我的证据链。"

### 面向用户

| 用户 | 关注点 | 网站策略 |
|------|--------|----------|
| **考研复试导师** | 学术潜力、工程素养、学习能力、独立思考 | 重点展示：设计决策过程、技术方案对比、状态机/算法设计、文档规范 |
| **校招面试官** | 落地能力、团队协作、代码质量、解决问题 | 重点展示：项目架构、代码片段、测试体系、性能优化、安全意识 |

| 维度 | 普通简历网站 | 本网站 |
|------|------------|--------|
| 内容深度 | 一句话介绍项目 | 每个项目含：背景、架构、核心设计、代码片段、测试结果、复盘 |

| 更新频率 | 一次性制作 | 作为长期技术成长记录持续更新 |

# 第四阶段：技术方案

---

## 推荐技术方案

### 方案对比

| 方案 | 前端 | 后端 | 数据库 | 适合度 |
|------|------|------|--------|--------|
| **A（推荐）** | Vue 3 + Vite 5 静态生成 | 不需要 | 不需要 | ⭐⭐⭐⭐⭐ |
| B | Vue 3 + Vite 5 | Spring Boot | MySQL | ⭐⭐ |
| C | Next.js / Nuxt | 不需要 | Markdown 文件 | ⭐⭐⭐ |
| D | GitHub Pages + Jekyll/Hexo | 不需要 | 不需要 | ⭐⭐ |

### 推荐：方案 A — Vue 3 静态站点

**理由：**

1. **你已经会 Vue 3**（江南出行用了 41 个页面），不需要学新框架
2. **静态站点足够**：个人网站本质是展示型网站，不需要后端和数据库
3. **部署成本最低**：GitHub Pages / Vercel / Netlify 免费托管
4. **维护成本最低**：无服务器、无数据库、无运维
5. **响应速度最快**：纯静态文件，CDN 加速

**具体技术选型：**

| 层级 | 技术 | 原因 |
|------|------|------|
| 框架 | Vue 3 + Composition API | 已有经验 |
| 构建 | Vite 5 | 已有经验 |
| UI 库 | Element Plus | 已有经验，适合内容密集型页面 |
| 动画 | CSS transitions + GSAP（按需） | 设计质量要求 |
| 内容管理 | Markdown 文件（项目内容）+ Vue 组件（布局） | 简单、Git 版本控制、易维护 |
| 代码高亮 | Shiki / Prism | 展示代码片段 |
| 图表 | ECharts 6 | 已有经验，技术栈展示 |
| 部署 | GitHub Pages / Vercel | 免费、自动部署 |
| 域名 | 可选：GitHub Student Pack 免费域名 | 学生福利 |
| 分析 | Google Analytics / Umami（自托管） | 了解访问情况 |

### 为什么不需要后端

- 没有用户系统、没有动态内容、没有数据提交
- 项目内容写 Markdown，Git 管理版本
- 简历 PDF 静态文件
- 联系方式用 `mailto:` 或表单服务（Formspree）

### 为什么不需要数据库

- 所有内容是静态的
- 不需要存储用户数据
- 后续如果加"博客评论"可以用 Giscus（基于 GitHub Discussions）

### 是否需要后台管理系统

**不需要。** 内容更新通过编辑 Markdown 文件 + Git Push 即可。这本身就是"工程师的工作方式"——用 Git 管理内容。

### 部署方案

```
GitHub Repo → Push → GitHub Actions → Build (Vite) → Deploy to GitHub Pages / Vercel
```

Vercel 推荐（支持自动部署、预览环境、自定义域名、免费 SSL）。

### 后期维护成本

| 维护项 | 频率 | 工作量 |
|--------|------|--------|
| 项目内容更新 | 每完成一个项目 | 30 分钟（写 Markdown + push） |
| 技术栈更新 | 每季度 | 15 分钟 |
| 博客文章 | 每周一篇 | 1-2 小时 |
| 依赖升级 | 每月 | 10 分钟（`npm update`） |
| 简历更新 | 每学期 | 20 分钟 |

**年度维护时间：约 50-80 小时**（含写博客），对于一个学生来说完全可承受。

## 总结：关键成功因素

### 必须做到的

1. **真实**：只展示你真做过的事，不包装不存在的能力
2. **深度优先**：宁可 3 个项目各有 3 个深挖点，不要 10 个项目只有名称
3. **可验证**：代码片段可运行、架构图有依据、数据有测试支撑
4. **持续更新**：网站是活的，不是一次性作品

### 避免的陷阱

1. ❌ 花哨但空洞的动画（导师和面试官想要看到的是内容深度，不是前端炫技）
2. ❌ 夸大技术能力（分布式锁在单机环境验证了就说明是单机环境）
3. ❌ 复制粘贴简历（网站应该是简历的"证据"，不是简历的 HTML 版）
4. ❌ 追求完美导致拖延（MVP 先上线，V1、V2 逐步完善）

### 下一步行动

1. **确认方案**：审阅本报告，确认网站方案和技术选型
2. **MVP 开发**：按 2 周目标开始开发
3. **内容准备**：整理 3 个项目的展示内容（架构图、代码片段、测试数据）
4. **代码 Review**：MVP 完成后进行代码 Review 和安全检查
5. **正式上线**：部署到 Vercel/GitHub Pages
