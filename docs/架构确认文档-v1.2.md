## 2. 确认移除 Element Plus

### 2.1 决策

**确认不使用 Element Plus 或任何第三方 UI 库。**

### 2.2 理由复审

| 理由 | 详细说明 |
|------|---------|
| **避免模板感** | Element Plus 的默认样式已被大量后台系统使用，面试官一眼能认出"又是 Element 写的" |
| **组件量极少** | 本网站的 UI 组件：导航栏、卡片、时间线、折叠面板、代码块、标签、按钮 — 手写 < 200 行 CSS 即可覆盖 |
| **设计控制力** | 手写组件可以精确控制每个像素，实现 Developer Academic 风格，而非 UI 库的通用风格 |
| **Bundle 体积** | 不引入 Element Plus 省 ~1MB+（gzip 后 ~200KB+） |
| **学习价值** | 手写组件本身就是"前端基本功"的证明 |

### 2.3 替代方案

| 需要的能力 | Element Plus 组件 | 手写替代 |
|-----------|------------------|---------|
| 导航栏 | el-menu | `<nav>` + CSS Flexbox |
| 卡片 | el-card | `<article>` + CSS 阴影/圆角 |
| 时间线 | el-timeline | `<ol>` + CSS `::before` 伪元素 |
| 折叠面板 | el-collapse | `<details>` + `<summary>`（原生 HTML） |
| 标签/徽章 | el-tag | `<span>` + CSS 圆角 |
| 按钮 | el-button | `<button>` + CSS 设计令牌 |
| 图标 | el-icon | Lucide Vue |
| 暗色模式 | — | CSS 自定义属性 + `data-theme` |

**关键原则：能用原生 HTML 元素的就用原生，不造轮子。** `<details>/<summary>` 折叠面板自带无障碍支持，比任何 UI 库都简洁。

## 4. 新增「技术决策」模块

### 4.1 设计意图

每个项目详情页底部新增「技术决策」章节，展示该项目中 **2-4 个关键决策**。

**不是"我做了什么"而是"为什么这样做而不那样做"。**

这是区别于普通简历网站的核心差异化内容——导师和面试官最想看到的就是"决策能力"。

### 4.2 展示格式

```
┌──────────────────────────────────────────┐
│  ## 技术决策                               │
│                                          │
│  ┌──────────────────────────────────────┐│
│  │ 决策 1 / 3                           ││
│  │                                     ││
│  │ 问题：订单并发调度如何避免重复分配？    ││
│  │                                     ││
│  │ 选项 A：数据库行锁                   ││
│  │   ✅ 实现简单                        ││
│  │   ❌ 数据库压力大、无超时机制         ││
│  │                                     ││
│  │ 选项 B：Redis 单锁                  ││
│  │   ✅ 性能好                         ││
│  │   ❌ 只锁订单不锁司机，司机可能被重复  ││
│  │                                     ││
│  │ 选项 C：两阶段分布式锁（✅ 选择）     ││
│  │   ✅ 零重复分配、自动超时释放         ││
│  │   ❌ 实现复杂度高、需要 Redisson     ││
│  │                                     ││
│  │ 选择理由：100 订单 × 20 司机压测     ││
│  │ 验证零重复，延迟 < 150ms             ││
│  │                                     ││
│  │ [相关代码 →]                        ││
│  └──────────────────────────────────────┘│
└──────────────────────────────────────────┘
```

### 4.3 三个项目的技术决策项

| 项目 | 决策项 |
|------|--------|
| **江南出行** | ① 并发调度：数据库行锁 vs Redis 单锁 vs 两阶段分布式锁 |
| | ② 司机评分：静态加权 vs 动态学习反馈（收敛型引擎） |
| | ③ 订单状态：if-else vs 状态机（形式化枚举） |
| | ④ 订单审计：日志表 vs 事件溯源 |
| **两地书 Love** | ① AI 集成：客户端直调 vs 云函数中转（安全考量） |
| | ② 数据访问：客户端直写 vs 云函数代理（安全分层） |
| | ③ 软删除：标记字段 vs per-user 数组（双人共用） |
| **题库系统** | ① 架构：单文件 HTML vs 工程化拆分（YAGNI 决策） |
| | ② 搜索：indexOf 线性扫描 vs 倒排索引（数据量考量） |

## 5. 新增「AI 工程实践」页面

### 5.1 设计意图

三个项目均使用 AI 辅助开发。与其让面试官问"你是不是全靠 AI"，不如主动展示 AI 工程实践流程。

### 5.2 URL 与定位

**URL:** `/ai-practice`
**导航名:** 「AI 实践」

### 5.3 页面结构

```

│  # AI 辅助软件工程实践                     │
│  > 不是"AI 帮我写代码"，                   │
│  > 是"我用 AI 加速了哪些环节，              │
│  >  我独立完成了哪些决策"                   │
│                                          │
│  ## 我的 AI 开发流程                       │
│  ┌──────────────────────────────────────┐│
│  │ 需求分析 ──→ 方案设计 ──→ 编码实现     ││
│  │  [人]        [人+AI]     [AI辅助]    ││
│  │                                 ↓    ││
│  │ 部署上线 ←── 代码审查 ←── 测试验证     ││
│  │  [人]        [人+AI]     [人]        ││
│  └──────────────────────────────────────┘│
│                                          │
│  ## AI 工具使用情况                        │
│  | 工具 | 用途 | 项目 |                    │
│  |------|------|------|                   │
│  | Claude Code | 架构设计、代码生成 | 全部 |│
│  | Trae | IDE 辅助 | 全部 |               │
│  | DeepSeek API | 业务 AI 功能 | 两地书 | │
│                                          │
│  ## 人机分工边界                           │
│  - 🧠 人做决策：架构选择、安全设计、算法设计 │
│  - 🤖 AI 辅助：代码生成、文档撰写、测试用例  │

│                                          │
│  ## 关键案例                               │

│                                          │
│  ### 案例 2：订单状态机                    │
```

## 6. Release Audit 阶段

### 6.1 定位

在 Task 006（部署与上线）之后，增加 **Task 007：Release Audit**，作为 MVP 的最后一道质量关卡。

### 6.2 Audit 检查清单

#### 内容审查

- [ ] 所有项目页内容与源代码一致（不夸大、不虚构）
- [ ] 所有指标数据可追溯来源（源文件数、API数、测试数）
- [ ] 无硬编码凭据（API Key、Token、AppID、密码）
- [ ] 无个人隐私信息（姓名以外的 PII、学号、手机号、身份证号）
- [ ] 代码片段不包含敏感信息
- [ ] 简历 PDF 不含过度个人信息
- [ ] Open Graph 图片不含隐私内容

#### 技术审查

- [ ] `npm run build` 零错误
- [ ] `npm run typecheck` 零错误（strict 模式）
- [ ] Lighthouse 评分 > 90（Performance / Accessibility / Best Practices / SEO）
- [ ] 所有路由刷新不 404
- [ ] 所有内部链接无死链
- [ ] 所有外部链接（GitHub）可访问
- [ ] 移动端（375px / 768px / 1024px）正常浏览
- [ ] 暗色模式所有页面正常
- [ ] 简历 PDF 可下载
- [ ] 页面加载 < 3 秒（Fast 3G 模拟）

#### 安全审查

- [ ] `.env` 和 `.env.local` 在 `.gitignore` 中
- [ ] `vercel.json` 不含密钥
- [ ] 无硬编码 API 端点指向内部服务
- [ ] CSP 头配置（Vercel 或 `_headers` 文件）
- [ ] 第三方字体加载使用 `font-display: swap`

#### 展示质量审查

- [ ] 三个项目详情页各有至少 1 张架构图
- [ ] 代码片段有语法高亮且无截断
- [ ] 面试准备页问题可折叠/展开
- [ ] 首页 3 秒内传达网站定位
- [ ] 简历下载按钮醒目
- [ ] Open Graph 分享预览正常

#### 交叉验证

- [ ] 请一个人（同学/朋友）浏览全站，记录困惑点
- [ ] 用"考研复试导师"视角浏览全站，记录是否找到想要的信息
- [ ] 用"校招面试官"视角浏览全站，记录是否找到想要的信息
