## 决策 1：单文件 vs 模块化架构（YAGNI 决策）

**问题：** 14,000+ 行的单文件是否需要拆分为多个文件？

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: 保持单文件** ✅ | 双击即用、零构建、易分发 | 编辑体验差、无模块化 |
| B: 拆分为 index.html + style.css + app.js + data.js | 编辑体验好 | 增加构建步骤、用户不能直接打开 |

**选择理由：** 核心约束是"打开即用"——用户双击 HTML 即可刷题。拆分文件意味着需要服务器或构建工具，违背了核心约束。清晰的章节分隔和命名规范使 14,000 行单文件实操可维护。

**不足：** 无基于导入的依赖图、无按文件区分的 linting。如果未来需要协作开发，拆分是必要的。

## 决策 2：线性搜索 vs 倒排索引

**问题：** 6000 道题的关键词搜索，用什么方案？

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: indexOf 线性扫描** ✅ | 零依赖、~1-3ms/次 | 题量暴增时性能下降 |
| B: 倒排索引 (lunr.js/Fuse.js) | 适合大文本量 | 增加 25-50KB 依赖 |

**选择理由：** 6000 题 × 200 字符 = 120 万字符，现代 JS 引擎中 indexOf 约 1-3ms。搜索频率低（每 session < 10 次），不值得为倒排索引付出额外依赖和维护成本。异步分批索引构建 + 150ms 防抖 + 结果上限 30 条确保体验流畅。

**不足：** 题库增长到 50,000+ 题时需要切换方案。升级路径明确：分词倒排索引 + Web Worker。

## 决策 3：深色模式 — CSS 变量 vs 重复选择器

**问题：** V8 使用 43 个重复选择器实现深色模式，维护困难。

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 43 个重复选择器 (V8) | 无 | 重复、遗漏、难维护 |
| **B: CSS 自定义属性** ✅ | 1 个 dark 块覆盖 20 个变量 | 需重构 CSS |

**选择理由：** `:root` 声明 65 个变量，`body.dark` 只覆盖 20 个颜色变量。所有组件自动适配深色模式。未来支持更多主题只需新增类 + 覆盖变量。

**不足：** 需要一次性重构（V8→V9 已完成）。
