## 决策 1：并发调度 — 两阶段分布式锁

**问题：** 多订单并发派单时，如何保证一个司机不被重复分配给两个订单？

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 乐观锁 (version 字段) | 实现简单 | 碰撞率高时重试成本大 |
| B: SELECT ... FOR UPDATE | 数据库层面互斥 | 行锁性能差、死锁风险 |
| **C: Redisson 二阶分布式锁** ✅ | tryLock 超时防死锁、fallback 防阻塞 | 依赖 Redis 单点、实现复杂 |

**选择理由：** 锁顺序统一 (order → driver) 数学上无死锁；tryLock 500ms 超时 + fallback 机制；100 订单 × 20 司机压测验证零重复分配。

**不足：** Redis Cluster 跨分片锁语义需额外验证；driver lock 伴随订单全生命周期，持有时间长。

## 决策 2：订单状态管理 — 枚举化状态机

**问题：** 出行订单有 10 个状态、15+ 合法流转路径。散落的 if-else 导致规则不可见、不可测试。

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: if-else 散落各处 | 无额外抽象 | 不可维护、不可测试 |
| B: 状态模式 (State Pattern) | 符合 OOP 原则 | 类爆炸（每个状态一个类） |
| **C: 枚举 + ALLOWED_TRANSITIONS Map** ✅ | 集中管理、易测试 | 不支持条件化流转 |

**选择理由：** 18 个单元测试覆盖全部合法/非法路径；新增状态改 1 个文件；新增流转路径加 1 行配置。

**不足：** "超时未接单"等条件化流转需要额外逻辑辅助。

## 决策 3：评分引擎 — 可插拔接口

**问题：** 调度策略需要从静态加权平滑演进到动态学习，且不能重构现有代码。

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: ScoringEngine 接口 + @Primary 注入** ✅ | 装饰器模式、切换只需改注解 | 状态存储在内存 |
| B: 策略工厂 (Strategy Factory) | 运行时灵活切换 | 过度设计 |
| C: 硬编码评分逻辑 | 简单 | 无法演进 |

**选择理由：** DynamicScoringEngine 包装 WeightedScoringEngine（装饰器模式）复用静态评分作为基础分；Spring 自动注入，从 Weighted 切到 Dynamic 只需改 `@Primary`。

**不足：** 评分状态存储在 ConcurrentHashMap，重启丢失。生产环境需持久化到 Redis。

## 决策 4：审计追踪 — 事件溯源

**问题：** 订单经历 8+ 状态变更，出问题后如何在 30 秒内定位根因？

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 订单表加 20 个时间戳字段 | 查询简单 | 字段爆炸、无法记录中间态 |
| B: 操作日志表 | 记录"谁做了什么" | 不能重建状态、统计需联表 |
| **C: 事件溯源表 (t_order_event)** ✅ | 可回溯、可统计、可重建 | 同步写入多一次 INSERT |

**选择理由：** 多种事件类型覆盖全流程；一条 SQL 回溯全生命周期；"系统派单 vs 司机接单"统计无需联表。

**不足：** 同步写入增加延迟。后续应升级到 Kafka 异步消费。
