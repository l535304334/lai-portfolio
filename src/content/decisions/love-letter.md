## 决策 1：AI 集成 — 云函数代理 vs 客户端直调

**问题：** 如何集成 DeepSeek API 而不暴露密钥、不失控成本？

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 客户端直调 | 简单、无冷启动 | API Key 暴露在 bundle 中 |
| **B: 云函数代理** ✅ | Key 仅服务端、可限流、可审查 | 冷启动 200-500ms 延迟 |

**选择理由：** 云函数中转实现了三层防护——频率限制（10次/分/用户）、内容审查（msgSecCheck）、Prompt 注入防护（边界标记）。共享模块消除 300+ 行重复代码。

**不足：** 冷启动延迟；频率限制 DB 故障时 fail-open（允许所有调用）。

## 决策 2：数据访问 — 客户端直写 vs 云函数代理（安全分层）

**问题：** 前端直接写数据库（快、简单）还是通过云函数（安全、可审计）？

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 全部客户端直写 | 快、无冷启动 | 安全规则不可从代码审计 |
| B: 全部云函数代理 | 最安全 | 高频操作受冷启动影响 |
| **C: 混合分层** ✅ | 敏感数据走云函数、高频操作走客户端 | 两类规则，维护复杂度增加 |

**选择理由：** `manageRecord` 云函数作为通用 CRUD 网关（集合白名单 + 字段白名单 + 创建者校验）；4 个客户端直写集合（打卡/课程/通知设置）是低风险 + 高频操作，依赖 CloudBase 控制台安全规则。

**不足：** 4 个客户端集合的安全规则不可从代码审计（非 IaC）。v1.1 计划全量迁移。

## 决策 3：软删除 — per-user 数组 vs 标记字段

**问题：** 双人共用场景——一人删除书信，另一人仍应看到。

**候选方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: deleted 布尔字段 | 简单 | 一人删、两人都看不到 |
| B: 独立删除映射表 | 干净 | 增加 JOIN、多一次读取 |
| **C: deletedBy: string[] 数组** ✅ | 零额外基础设施 | 旧数据累积 |

**选择理由：** 每人独立删除不干扰对方；查询简单（`deletedBy: _.nin([userId])`）；与 `readBy` 数组模式统一。数组长度上限为 2（情侣），无扩展问题。

**不足：** 旧删除数据在 DB 中累积（可加定期清理任务）。
