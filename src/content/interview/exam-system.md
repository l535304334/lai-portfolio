### Q1: 14,000 行单文件 JS 应用，没有模块系统，怎么保持可维护性？

**考察点：** 代码组织、自文档化、对无工具环境的实用性理解

**回答思路：**
1. 按功能区域严格分区（搜索/SM-2/报告/考试），清晰注释分隔
2. 所有状态变量在文件顶部统一声明
3. 一致的命名规范：动词+名词（buildSearchIndex、toggleReviewMode、handleOptionClick）
4. 设计令牌系统将样式逻辑解耦
5. 单文件约束是用户需求驱动的（零配置启动），不是缺乏规划能力

**关键词：** 代码组织、YAGNI、状态集中管理、命名规范

### Q2: 6000 道题的搜索用 indexOf，考虑过倒排索引吗？

**考察点：** 性能意识、数据规模决策、实用主义 vs 过早优化

**回答思路：**
1. 测量：6000 × 200 字符 ≈ 120 万字符，indexOf ~1-3ms
2. 搜索频率低（< 10 次/session），不值得为倒排索引付出内存和维护开销
3. 150ms 防抖 + 结果上限 30 条 + 异步索引构建
4. 升级路径明确：题量 50,000+ 时引入分词倒排索引 + Web Worker

**关键词：** 性能测量、过早优化、防抖、分批构建

### Q3: SM-2 算法与标准 SM-2 的区别？如何与错题系统集成？

**考察点：** 间隔重复理论理解、系统集成设计

**回答思路：**
1. 标准 SM-2 用 0-5 评分 → 我简化为 2（错）和 5（对），因为医学题是二元的
2. EF 公式与标准一致：`EF' = EF + (0.1 - (5-q)*(0.08 + (5-q)*0.02))`
3. 集成策略：wrongSet 保持原义（错题集合），SRS 在其上叠加调度
4. 复用 applyFilter() 添加 reviewMode 分支，最小侵入

**关键词：** SM-2、EF 因子、间隔调度、最小侵入集成

### Q4: 如何解决 localStorage 配额问题？

**考察点：** 客户端存储约束、防御性编程

**回答思路：**
1. 三层防御：压缩（快照去 questions 字段节省 90%）→ 自动淘汰（最旧快照）→ 主动监控（navigator.storage.estimate()）
2. lsSet 的 try/catch + evictOldestSnapshot() + 重试
3. 10MB 阈值预警（Chrome 上限）

**关键词：** 配额管理、try/catch 重试、storage.estimate()、快照淘汰

### Q5: 如果要求支持多人使用，架构怎么改？

**考察点：** 全浏览器端架构的局限性、后端基础理解

**回答思路：**
1. 坦诚：当前设计不为多人——localStorage 是单机的
2. 最小方案：IndexedDB + 导出/导入共享（已支持）
3. 真正多人：后端（Node.js/Express 或 Firebase），数据绑定用户
4. 当前架构天然适合"同宿舍共享 JSON 文件"——零成本多人协作

**关键词：** 架构局限、渐进增强、CRDT、离线优先
