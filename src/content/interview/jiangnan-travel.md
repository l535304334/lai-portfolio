### Q1: 你这个项目最难的地方是什么？

**考察点：** 分布式系统理解 + 并发问题解决能力

**回答思路：**
1. 并发调度一致性 — 50 订单 × 20 司机同时竞争，synchronized 无法跨 JVM
2. 方案：二阶分布式锁（order lock → driver tryLock 500ms → fallback）
3. 三个关键细节：锁顺序 100% 统一保证无死锁；加锁后重读 DB 防 ABA；tryLock 防活锁
4. 压测结果：100 订单 × 20 司机，零重复分配、零死锁

**关键词：** Redisson、二阶锁、tryLock、ABA 竞态、WatchDog

### Q2: 为什么设计 order_event 表而不是用操作日志？

**考察点：** 架构设计思维 + 数据建模能力

**回答思路：**
1. 这不是日志，是轻量级事件溯源（Event Sourcing）
2. 三个作用：审计追踪（回溯全生命周期）、状态重建（不需要 20 个时间戳字段）、统计分析（COUNT + GROUP BY 直接出报表）
3. 为未来 Kafka 异步消费和 CQRS 架构铺路

**关键词：** 事件溯源、CQRS、审计追踪、12 种事件类型

### Q3: 评分引擎怎么保证公平性？

**考察点：** 系统设计中的收敛性和公平性问题

**回答思路：**
1. 问题：静态评分产生正反馈循环，高分司机垄断订单
2. 方案：动态学习层（接单 +5/拒单 -10 + 10 分钟半衰期 + 上下限 [-30, +20]）
3. 为什么收敛：拒单多的排名下降 → 空闲久的被推上来 → modifier 自然回归 0
4. 量化结果：5 轮从 50% 收敛到 70%

**关键词：** 正反馈循环、动态评分、时间衰减、收敛性

---

### Q4: 有遇到分布式锁的坑吗？

**考察点：** 分布式系统工程经验 + 从 bug 中学习的能力

**回答思路：**
1. 核心坑：ABA 竞态条件 — tryLock 成功后未重读 DB，读到过期状态
2. 修复：加锁后立即 `driverMapper.selectById()` 重新检查
3. 教训：分布式锁只保证互斥，不保证数据新鲜度
4. AI 生成的代码在这里有个 bug，人工审查发现的

**关键词：** ABA、重读 DB、WatchDog 续期、finally 释放

### Q5: 如果重新设计评分引擎，怎么做？

**考察点：** 系统演进思维 + 架构可扩展性

**回答思路：**
1. 当前瓶颈：findCandidates 全表扫描
2. 优化：Redis Sorted Set 实时维护司机排名，O(logN) 查询
3. 关键：不需要大改 — ScoringEngine 是接口，替换实现即可
4. 还可以接入真实运营数据做 A/B 测试

**关键词：** Sorted Set、可插拔接口、A/B 测试
